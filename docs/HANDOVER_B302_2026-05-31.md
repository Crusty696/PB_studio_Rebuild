## 1. Was B-302 ist (aus Bug-File, Original-Befund 2026-05-11)

App-Crash beim Klick auf Auto-Edit-Preset:

```
TypeError: 'PySide6.QtWidgets.QProgressBar.setValue' called with wrong argument types:
  PySide6.QtWidgets.QProgressBar.setValue(str)
→ Hard-Kill Exit -1073741819 = 0xC0000005 = STATUS_ACCESS_VIOLATION
→ Faulthandler: rekursive perf_watchdog._profiled_notify Frames (Slot-in-Slot Reentrancy → Heap-Korruption)
```

**Kausalkette laut Bug-File:** overloaded `progress`-Signal mit Default-Overload
`(str, float)` + nackter `.connect` (ohne Overload-Selector) → Slot bekommt
`pct=<stage_key:str>`, `msg=<fraction:float>` → `task.progress = "audio_load"`
→ `progress_bar.setValue(str)` → TypeError aus Qt-C++-Kontext (kein Python-except)
→ bei Wiederholung + paralleler GPU-Last → Reentrancy → Access-Violation.

**Bug-File nennt 2 dokumentierte Fixes (2026-05-11):**
1. `worker_dispatcher.py`: progress-Slot toleriert beide Overloads + coerced `pct` zu `int`.
2. `task_manager.py update_task`: defensive `int()`-Coercion für `progress`+`total`.

**Status laut Bug-File widersprüchlich** → siehe §4.

## 3. RESTRISIKO — was eine Live-Verifikation noch klären sollte

> Die Crash-Kette ist statisch geschlossen (§V9, 16 Tests). Diese Punkte sind
> Rest-Sorgfalt, kein offener Blocker:

1. **Original-Crash war NATIVE `0xC0000005` unter Laufzeit-Last** (rekursiver
   `perf_watchdog._profiled_notify` bei wiederholtem Auto-Edit + paralleler

   0 `[CRITICAL]`). (Regressionstest `test_b323` deckt nur den TypeError-Auslöser
   statisch ab, nicht die native Reentrancy unter GPU-Last.)

   `update_task` einen `str` halten (V7). Aktuell harmlos (Konsument coerced),
   aber andere künftige Konsumenten von `task.progress` müssten `_as_int` auch
   nutzen. Defensive int()-Coercion in `update_task` (= dokumentierter Fix #2)
   wäre Gürtel-und-Hosenträger — **nicht nötig für B-302, nicht eigenmächtig**.
3. **Bug-File-Doku stale:** beschreibt Fix #1/#2 als die Lösung; tatsächlich
   wirkt `_as_int`. Doku angleichen (siehe §4).
